# Release v1.4.0 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Cut release v1.4.0 ŌĆö bump the version, write the CHANGELOG entry and README badge, push to `main`, then push tag `v1.4.0` which triggers `build-release.yml` to produce a **draft** GitHub Release with a DMG.

**Architecture:** Pure release procedure, no app code change. Two files carry the version (`core/version.py`, `pyproject.toml`) and are kept in sync by `tests/test_version.py`. The release is one `release: v1.4.0` commit on `main` (version + CHANGELOG + README badge), pushed; then an annotated `v1.4.0` tag on that commit, pushed, which is the sole trigger of `build-release.yml`. The agent does NOT publish the draft release ŌĆö that is the user's manual outward action.

**Tech Stack:** Python 3.12, git, GitHub Actions. Env: pytest/ruff = `.venv/bin/python -m pytest` / `.venv/bin/python -m ruff` (system python is 2.7; deps in `.venv`). Pre-commit hook runs ruff + ruff format + pytest on staged `.py` files ŌĆö let it run, never `--no-verify`. Design doc: `docs/plans/2026-05-18-release-v1-4-0-design.md`.

**Preconditions:** On branch `main`, clean working tree, `main` in sync with `origin/main`. Current version `1.3.3`. No remote tag `v1.4.0` exists.

```markdown
## v1.4.0 ŌĆö 2026-05-18 (App-activation catch-up & update check)

### Added
- **Missed daily check now caught up on app activation.** Paragraphos
  runs continuously in the tray, but the daily feed check only fired at
  the scheduled time and was lost if the Mac was asleep/off, the app was
  busy, or the check failed (offline). It is now re-run the next time the
  app is brought to the foreground ŌĆö gated to once per missed slot via
  the existing `should_catch_up` logic, with a re-entrancy latch so a
  rapid refocus during the start delay can't double-fire or emit a
  spurious "already running" toast. A failed/offline/stopped check no
  longer falsely marks the slot done, so it is genuinely retried.
- **Periodic update check on app activation.** The GitHub-release update
  check previously ran only once at startup, so a long-lived tray session
  never noticed a new release. It now also re-checks when the app is
  foregrounded, gated to at most once per 24 h, fully decoupled from the
  catch-up path. A new **Settings ŌåÆ "Check for updates"** toggle (on by
  default) governs both the startup and activation checks; off means zero
  GitHub requests.
- **Update notification deduped per release.** The tray "update
  available" message now fires once per release tag instead of on every
  launch / every check while an update is pending; the in-window banner
  remains the persistent reminder.

### Fixed
- **Discovery routed through the shared httpx client.** iTunes podcast
  search and cover-art fetching now use the same pooled `httpx` client
  as the rest of the app instead of ad-hoc requests ŌĆö consistent
  timeouts, headers, and connection reuse.
- **Worker orphan-claim scoped to the run-start snapshot.** The queue
  worker could reclaim episodes added after a run began, pushing the
  progress counter past the total (`done_idx > total`). Orphan-claim is
  now bounded to the snapshot taken at run start, so `done_idx Ōēż total`
  always holds.
```

### Task 5: Monitor the build & report

**Files:** none ŌĆö observation only.

**Step 1: Poll the build-release run to completion**

The `macos-14` build (py2app + DMG) takes several minutes. Poll roughly every ~3ŌĆō4 minutes (cache-friendly) until `status` is `completed`:
```bash
curl -s -m 15 "https://api.github.com/repos/madevmuc/paragraphos/actions/workflows/build-release.yml/runs?per_page=1" | python3 -c "import sys,json; d=json.load(sys.stdin); r=d['workflow_runs'][0]; print(r['head_branch'], r['status'], r['conclusion'], r['html_url'])"
```

**Step 2: On `completed`**
- If `conclusion == success`: fetch the draft release URL:
  ```bash
  curl -s -m 15 "https://api.github.com/repos/madevmuc/paragraphos/releases?per_page=3" | python3 -c "import sys,json; d=json.load(sys.stdin); [print(x['tag_name'], 'draft' if x['draft'] else 'published', x['html_url']) for x in d]"
  ```
  Report the `v1.4.0` draft release URL to the user. Remind them: it is a **draft** ŌĆö they review and publish it manually; publishing is what flips `releases/latest` and triggers user update notifications. **Do not publish it.**
- If `conclusion == failure`: fetch the failing job + annotations (same `check-runs` API approach used previously), report the error. Recommend (do not auto-execute) tag rollback if the failure is in the release itself: `git push --delete origin v1.4.0 && git tag -d v1.4.0`. Ask the user before any rollback.

**Step 3: Done** ŌĆö report final state: release commit SHA on main, tag pushed, build conclusion, draft release URL (or failure detail).

## Notes

- **No `--no-verify`, no force-push, no tag re-push.** A botched tag is rolled back explicitly (delete remote + local) only after asking the user.
- **Agent never publishes the GitHub Release.** Draft ŌåÆ user publishes manually (conscious outward action that triggers end-user update notifications).
- DRY: version lives in exactly two files kept in sync by `tests/test_version.py`; no other file hardcodes it (setup*/about import the symbol).
- YAGNI: no release-notes automation beyond the workflow's `generate_release_notes`; CHANGELOG is the curated user-facing record.
